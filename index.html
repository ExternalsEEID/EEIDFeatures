Token lifetime & session controls
What each control does — see Workforce vs External ID tab for availability
Access token lifetime
A short-lived credential your app uses to call APIs on behalf of the user. Once it expires, the app must get a new one silently using the refresh token — or prompt the user to sign in again if no refresh token is available. Setting this to 60 min means the token is valid for exactly 1 hour before it must be renewed.
ID token lifetime
Tells your app who the user is — contains identity claims (name, email, user ID). Typically used at sign-in to establish the app's own session. The app reads this once; it doesn't repeatedly call APIs with it. Lifetime controls how long this identity assertion is considered valid.
Refresh token lifetime
A longer-lived credential used to silently get new access tokens without asking the user to sign in again. Think of it as the "stay logged in" mechanism. When it expires, the user must fully re-authenticate. Not configurable in External ID — platform-managed.
Refresh sliding window — Bounded
An Azure AD B2C concept. "Bounded" means the refresh token can keep renewing itself (sliding) but only up to a maximum total duration — e.g. 90 days. After that hard limit, the user must sign in again regardless of activity. Prevents indefinite "stay logged in forever" sessions.
Web app session lifetime
An Azure AD B2C concept. Controls how long the browser cookie for the B2C session lasts — independent of the token lifetime. Once the session cookie expires, the user is redirected to re-authenticate even if their access token is still valid. Not a native Entra External ID setting.
CA sign-in frequency
A Conditional Access control that forces the user to actively re-authenticate after a set time (e.g. every 4 hours), even if their tokens are still valid and the refresh token hasn't expired. Adds a hard re-auth checkpoint on top of token mechanics. Supported in External ID.
Persistent browser session
Controls whether the user stays signed in after closing and reopening the browser. When enabled, a persistent session cookie is set — the user won't be prompted to sign in again on their next visit. When disabled, closing the browser ends the session. Supported in External ID.
Continuous access evaluation (CAE)
Instead of waiting for the access token to expire, CAE allows Entra to revoke access in near real time — e.g. the moment an account is disabled or a password is changed. The resource is notified immediately and rejects the token. Not available in External ID.
Token protection (token binding)
Cryptographically binds a token to the specific device it was issued to. Even if an attacker steals the token, they can't use it from a different device. Prevents token theft and replay attacks. Not available in External ID.

Key principle: External ID is a separate tenant from your workforce. Apps, users, branding, and user flows are all managed independently. Enterprise security features (ID Protection, ID Governance, PIM, device compliance) are not available — External ID is purpose-built for consumer and business-customer CIAM scenarios.
⚠ Security limitations & workarounds
Workforce security features not available in External ID — and what to do instead
Custom banned password list
Not available — Authentication methods → Password protection is workforce-only. Global banned list still applies automatically.
Workarounds
→ OnAttributeCollectionSubmit extension — validate against your own banned list at sign-up
→ Use Email OTP as primary — no password, no policy needed
→ Built-in complexity still enforced (8+ chars, mixed case, number/symbol)
ID Protection & risk detection
Not available — sign-in risk, user risk, risk-based Conditional Access, and risky user / sign-in reports are all workforce-only.
Workarounds
→ Akamai / Cloudflare via Microsoft Security Store for bot & DDoS protection
→ Custom auth extensions for risk logic at sign-up
→ Sign-up fraud prevention via partner solutions
ID Governance & PIM
Not available — entitlement management, access reviews, lifecycle workflows, and privileged identity management are workforce-only.
Workarounds
→ Workforce tenant + B2B collaboration if full governance is required
→ Graph API for custom user lifecycle automation
→ Time-limited access via user flow expiry settings
CAPTCHA (sign-in & sign-up)
No configurable CAPTCHA setting for sign-in or sign-up flows — this was an Azure AD B2C feature (user flow Properties → CAPTCHA) and has not been carried across to External ID. The only CAPTCHA in EEID is platform-managed: automatically triggered during SMS MFA when fraud is suspected.
Workarounds
→ OnAttributeCollectionSubmit extension — implement custom CAPTCHA (e.g. Google reCAPTCHA, hCaptcha) via your own API at sign-up
→ Akamai / Cloudflare via Microsoft Security Store — bot protection & WAF at network layer
→ MFA enforcement — adds a human verification step, reducing bot effectiveness
Report suspicious activity
Generally not available in External ID. This feature is tied to Microsoft Authenticator push notifications and voice call MFA as the delivery channel — when a user receives an unexpected prompt, they can flag it directly in the app or via phone. Since External ID primarily uses Email OTP, SMS OTP, and passkey (FIDO2) MFA, none of which support this interaction model, the experience has no delivery mechanism in External ID.
Exception
If users authenticate via a federated IdP (e.g. SAML/WS-Fed enterprise IdP) that issues Authenticator-based MFA prompts upstream, the report mechanism may be available at the IdP level — but this is outside External ID's own MFA stack and not configurable within the External ID tenant.

B2B direct connect
Allows external users to access shared resources without a guest account — blocked by default
What B2B direct connect is
Enables users from external tenants to access shared channels in Teams without creating a guest account
User remains in their home tenant — no object created in your directory
Different from B2B collaboration which creates a guest (#EXT#) account
Requires both tenants to configure inbound/outbound settings
Default: Blocked for all external orgs out of the box
Can be enabled per approved partner tenant via Organisational settings
B2B collaboration vs direct connect
B2B collaboration — external user invited as guest, object created in your tenant, managed via access reviews
B2B direct connect — no guest account, user stays in home tenant, less visibility and governance
B2B direct connect is higher risk — no object in your directory means no Entra ID governance controls apply
Cross-tenant sync — different again, syncs users as members (not guests) for seamless collaboration

Controls whether claims from external tenants are accepted — configurable per org
Trust MFA from Entra tenants

⚠
External ID scope: Cross-tenant access settings (B2B collaboration, B2B direct connect, trust settings) apply to workforce tenants only per official Microsoft documentation. The requirement to restrict default inbound/outbound access and disable B2B direct connect by default applies to your workforce tenant — not the External ID external tenant. External ID external tenants manage consumer/customer access through user flows, not cross-tenant B2B mechanisms.
